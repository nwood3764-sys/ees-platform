# EnergyPlus runner

A small container that exists only to run a raw OpenStudio `.osm` model: the
browser can't run EnergyPlus (a ~200 MB native binary needing real CPU and a
weather file). The front end POSTs an `.osm` here; the runner repairs the model,
runs EnergyPlus 25.2.0, reads the results, and returns JSON the front end drops
straight into its data model.

## Endpoints

| Method | Path        | Body                         | Returns |
|--------|-------------|------------------------------|---------|
| GET    | `/health`   | —                            | `200 "ok"` |
| POST   | `/run-osm`  | multipart, field `file`=.osm | JSON results |

Response JSON (same keys the front end's `parseOpenStudio()` produces):

```json
{
  "elec": [12 monthly kWh], "gas": [12 monthly therms], "name": "...",
  "elecEnd": {"Heating": kWh, ...}, "gasEnd": {"Water Systems": therms, ...},
  "elecMonthly": {"Heating": [12], ...}, "gasMonthly": {"Heating": [12], ...},
  "_notes": ["repair messages"]
}
```

## Why JSON, not eplustbl.htm

The build spec suggested returning `eplustbl.htm` to reuse the front-end parser.
That parser, however, targets the OpenStudio **Results reporting measure** HTML
(captions like `Electricity Consumption (kWh)`, toggles like `EUI - Electricity`),
which is generated by a Ruby measure run through the OpenStudio CLI. The
pip-installed `openstudio` SDK (pinned to 3.11.0) has no Ruby measure runtime and
cannot regenerate that HTML; native `eplustbl.htm` uses different headings the
parser would not match. So the runner reads the authoritative `eplusout.sql` and
emits the parser's **output shape** directly — the robust equivalent, with the
same downstream behavior (scaling, XML, xlsx). `eplustbl.htm` is still written to
the run dir for debugging.

## The two model repairs (required)

Asset-Score-exported `.osm` files fatally crash EnergyPlus without these
(`repair_and_run.py`):

1. **Broken `Schedule:File` ("water issue")** — a "SHW Ambient Temperature"
   `Schedule:File` points at a CSV not shipped with the `.osm`. We remove all
   `Schedule:File` and set each water heater's ambient temperature to a constant
   21 °C schedule.
2. **Incomplete recirculation `PlantLoop`** — a service-hot-water recirc loop is
   missing its operation scheme. We remove any plant loop named "recirc".

Both are applied identically to baseline and improved, so savings stay clean;
only absolute energy is marginally lower (and the app scales energy to the Asset
Score EUI anyway).

## Version lock (load-bearing)

OpenStudio SDK **3.11.0** ↔ EnergyPlus **25.2.0**. A mismatched engine (e.g.
24.1) fails the SDK-translated IDF with "Object contains more field values than
maximum". Do not bump one without the other.

Output requests are set through the SDK, never by text-appending to the IDF:
appending a second `OutputControl:Table:Style` when the model already has one is
fatal.

## Local testing (without Docker)

```bash
pip install -r requirements.txt --break-system-packages
./weather/fetch_weather.sh
AUDIT_EPW="$(pwd)/weather/USA_WI_Milwaukee-Mitchell.Intl.AP.726400_TMY3.epw" \
  python3 repair_and_run.py /path/to/model.osm        # prints results JSON
```

A clean EnergyPlus run prints `EnergyPlus Completed Successfully -- N Warning;
0 Severe Errors`. The runner checks for the literal `0 Severe Errors` phrase, not
a raw count of the word "Severe" (the summary lines contain it).

## Deploy (Fly.io, scale-to-zero)

```bash
fly launch --no-deploy        # creates the app; adjust fly.toml app name first
fly secrets set ALLOWED_ORIGIN="https://audit-template.ees-wi.org"
fly deploy
curl https://<app>.fly.dev/health     # -> ok
```

Then set `window.AUDIT_RUNNER_URL` in `../frontend/config.js` to
`https://<app>.fly.dev`.

`CORS` is restricted to `ALLOWED_ORIGIN` (comma-separated, or `*`). Railway is a
simpler always-on alternative if scale-to-zero isn't needed.
